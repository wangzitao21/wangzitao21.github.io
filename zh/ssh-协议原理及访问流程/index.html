SSH 协议原理及访问流程

ssh 即 Secure Shell，也就是在 shell 上增加了加密功能。我们常见的 Linux 服务器登录时都使用了 ssh。

之前只是糊涂地使用 ssh，并没有搞清楚里面的原理和具体步骤，这次系统地做了一些梳理。

不过本文仍旧比较简单，今后可能会随我认识的加深而进一步更新。

本节内容部分参考了以下文章:

在使用 ssh 访问服务器时，主要经历了以下五个流程:

1 TCP 握手和协商#

该过程又可以划分为以下三个部分:

  1. 建立 TCP 连接：客户端首先向服务器的监听端口（默认是 22 端口）发起 TCP 连接请求，并建立 TCP 三次握手连接（TCP Handshake）。
  2. 版本协商：连接建立后，客户端和服务器交换各自支持的 ssh 协议版本信息，并协商出共同支持的最高版本作为后续通信版本。
  3. 算法协商：客户端和服务器交换各自支持的加密算法、密钥交换算法、MAC 算法、压缩算法等列表，协商出一组双方都支持且最优的算法套件，用于后续的加密通信、认证和完整性校验等。

2 服务器身份验证#

一旦 TCP 连接建立，服务器会首先向客户端发送服务器的公钥/证书，即 host key，以便客户端对服务器进行身份验证（Server Authentication）。

host key 是用于验证服务器身份的非对称加密密钥对，包括一组公钥和私钥。host key 用于防止中间人攻击，确保客户端连接的是正确的服务器。这对 host key 会存储在服务端上，其位置为：

在连接时，服务器会发送自己的公钥 ssh_host_rsa_key.pub，这时客户端会进行如下形式的验证：

  • 若客户端首次连接到该服务器，客户端会收到警告，提示这是一个新的服务器，并询问是否信任该主机，显示类似 The authenticity of host 'xxx' can't be established 的提示。客户端一旦确认没问题，则会将服务器的公钥保存到客户端本地的 ~/.ssh/known_hosts 文件中，以便后续直接连接。
  • 若客户端以前连接过此服务器，客户端会检查收到的公钥是否与 known_hosts 中存储的公钥一致。一致则连接继续；不一致则可能有中间人攻击的风险，客户端会警告并拒绝连接。

如果是使用 ssh CA 签署的 host key，则可以免除此询问过程，客户端会直接信任服务器端。

由于 ssh 连接是由客户端发起的，所以上述步骤为了让客户端核实服务器是否为客户端要连接的正确服务器，以防止被中间人截胡。

3 密钥交换#

完成服务器身份验证后，客户端会和服务器端使用之前协商好的密钥交换算法（如最常用的 Diffie-Hellman 算法）生成一个共享的会话密钥（Session Key）。该过程即密钥交换（Key Exchange）。

Session Key 是对称加密密钥，加密和解密速度较快，用于加密接下来会话中的所有数据。此外，Session Key 是临时的，只在当前会话中有效，客户端和服务器同时持有，都使用这个 Session Key 来加密和解密数据。

Session Key 的生成步骤（Diffie-Hellman 算法）为：

  1. 服务端先生成两个预先约定好的数字 G 和 P。
  2. 服务端生成一个临时私钥 b，并使用 G, P 和 b 生成公钥 y。
  3. 服务端把 G, P 和 y 直接明文发送给客户端。
  4. 客户端生成一个临时私钥 a，并使用接收到的 G, P 与 a 计算出公钥 x。
  5. 客户端将公钥 x 发送给服务端。
  6. 服务端使用私钥 b 和客户端发来的 x, 以及 P 计算，得到 K。
  7. 客户端使用私钥 a 和服务端发来的 y, 以及 P 计算，得到 K。

这里的 K 就是 Session Key。

为什么服务端和客户端得到的都是 K？

已知: 客户端 K 的计算公式是 $K = y^a \bmod P$， 服务端 K 的计算公式是 $K = x^a \bmod P$,

而 x 和 y 计算公式是: $y = G^b \bmod P$， $x = G^a \bmod P$，

所以可得: $y^a \bmod P = (G^b \bmod P)^a \bmod P = G^{(b\times a)} \bmod P$， $x^b \bmod P = (G^a \bmod P)^b \bmod P = G^{(a\times b)} \bmod P$，

因此二者计算的 K 相同。

4 用户身份验证#

完成密钥交换后，服务端开始验证客户端身份（User Authentication）。

我们早先在部署一台服务器时，会先在客户端本地生成一对非对称密钥，其中私钥由客户端自己保留，公钥粘贴到服务端的 /home/<user>/.ssh/authorized_keys 里。

在用户验证阶段，服务端就通过这个公钥来验证客户端。具体而言，客户端会使用私钥将一段数据进行签名，并将签名发送给服务端。服务端使用客户端的 authorized_keys 中的公钥来验证签名，如果验证通过，则认证成功。

但是在这篇文章中，我看到客户端验证过程可能会更复杂一点 每天都在用 SSH，你知道 SSH 的原理吗？，即多了一个随机挑战内容，而不只是验证签名。大致可划分为这几步：

  1. 客户端发起认证请求：客户端使用自己的私钥对一个数据块进行签名（这个数据块通常包含会话 ID 等信息），并将签名和公钥 ID 发送给服务器。
  2. 服务器验证公钥：服务端在 authorized_keys 中找到客户端提供的公钥 ID 对应的公钥，并使用该公钥验证客户端的签名，确认是对应的客户端发起的请求。
  3. 服务器发起挑战：服务器会生成一个随机的字符串，称之为挑战，并使用客户端的公钥对字符串进行加密，然后将加密后的挑战发送给客户端。
  4. 客户端响应挑战：客户端使用自己的私钥解密服务器挑战，得到原始字符串。然后，为了进一步证明自己是实时客户端而不是重放攻击，客户端还会使用 Session Key 对字符串再加密，并将结果发送给服务器。
  5. 服务器验证响应：服务器使用相同的会话密钥对之前生成的挑战值进行加密，并将结果与客户端发送的结果进行比较。若两者一致，则认证通过，允许客户端登录。

挑战-响应机制的优势在于防止重放攻击（Replay Attack），换句话说，即使有人之前窃取了客户端的签名，也无法通过认证，因为每次服务器生成的挑战都是不同的。这样能确保当前进行认证的就是真正的客户端，而不是一个录制下来的认证过程。

另外，用户身份验证过程一般还可以使用密码验证，不过不安全，一般建议关闭。

5 建立加密会话#

最后，客户端和服务端就会开始用 Session Key 加密通信内容。所有后续的数据传输（命令、文件、输出等）都会使用会话密钥进行加密。
